# 설악산 재난·구조관리 플랫폼 — 현황 분석 및 개선 계획표

> 작성일: 2026-06-12 · 기준 코드: `www/index.html` (~7,600줄 단일 파일 PWA)

---

## 1. 현재 시스템 동작 요약

| 영역 | 현재 상태 |
|---|---|
| **구조 흐름** | 1보 등록 → 고도화 타임라인(팀 생성·표지판 통과·환자조우·하산/병원이송·복귀) → 추가보고(2보~) → 종료 |
| **지도** | 카카오맵 기반. 구조/위험 핀, 팀 위치 칩, 포커스 모드(경로 폴리라인 + 표지판 번호 + 차량 실경로·소요시간) |
| **데이터** | Firestore(rescues/hazards는 문서별, 나머지는 단일 JSON) + localStorage 캐시, 실시간 리스너 동기화 |
| **인증** | 카카오 OAuth + 승인코드 가입 승인제, 관리자 로컬 인증(마스터 계정은 운영자 요구로 유지) |
| **알림** | 앱 내 알림(notis) + 기기 간 sharedNotis(Firestore). FCM 푸시는 **미완성** (VAPID 미설정) |
| **외부 연동** | 기상청 API(현재날씨·특보), 카카오 모빌리티(차량 경로), Google Sheets webhook(선택) |
| **통계** | 구조/위험 통계 차트, 기간 선택 CSV 다운로드 |
| **시설물** | 점검 지도/목록/통계, 다목적위치표지판 → 구조 경로 자동 생성의 기반 데이터 |

## 2. 부족한 점 (현재 기능의 결함)

### 🔴 안정성 — 데이터 유실 위험
| # | 문제 | 위치/근거 |
|---|---|---|
| A1 | **Firestore 쓰기 실패가 조용히 무시됨** — `.catch(()=>{})` 가 40곳 이상. 산속 통신 불량 시 저장 실패해도 사용자는 모름 | `DB.s` 등 전역 |
| A2 | **오프라인 사진 큐가 메모리에만 존재** — 업로드 대기 중 앱 종료하면 사진 유실 | `_offlinePhotoQueue` |
| A3 | **동시 편집 충돌** — 두 기기가 같은 구조를 수정하면 마지막 저장이 전체를 덮어씀 (팀 통과 기록 유실 가능) | rescue 문서 단위 저장 |
| A4 | **삭제 캐스케이드 없음** — 구조 삭제 시 댓글(`comments_*`), Storage 사진이 고아로 남음 | `admDelRes` |
| A5 | SW 오프라인 캐시가 `/` 한 개뿐 — 오프라인 첫 화면도 보장 안 됨 | `sw.js _SHELL` |

### 🟠 보안
| # | 문제 |
|---|---|
| B1 | **Firestore 보안 규칙 미점검** — 설정값이 소스에 공개돼 있어 규칙이 느슨하면 누구나 환자 개인정보(이름·생년·병력) 읽기/쓰기 가능. **최우선 점검 필요** |
| B2 | 사용자 입력이 `innerHTML`에 들어가는 곳 다수에서 `_esc()` 미적용 → XSS 가능 (제목·이름·댓글 등) |
| B3 | 관리자 비밀번호가 무염(salt 없는) SHA256 — 무차별 대입에 취약 (마스터 계정 유지는 운영 요구사항이므로 해시 강화만) |
| B4 | 기상청·카카오 API 키 소스 노출 — 도용 시 쿼터 소진 (정적 호스팅 한계, 도메인 제한 설정으로 완화) |

### 🟡 미완성 기능
| # | 문제 |
|---|---|
| C1 | **FCM 푸시 미완성** — VAPID 키 미설정 + 발송 주체(서버) 없음. 앱이 꺼져 있으면 새 구조 발생을 알 방법이 없음 → **재난 앱의 핵심 결함** |
| C2 | `staff` 데이터 초기화 후 미사용 — 직원 명부와 팀원 선택이 단절 |
| C3 | `timetable`(상황 타임테이블)과 `wpLog`(표지판 통과 기록) 이중 체계 — 보고서에 통합 표시 안 됨 |
| C4 | 위키 이미지 미리보기 없음, 사진 1장 제한 |

## 3. 개선·신규 계획표

### Phase 0 — 긴급 (안정성·보안) · 권장 즉시
| 항목 | 내용 | 작업량 |
|---|---|---|
| **P0-1. 저장 실패 가시화 + 재시도 큐** | `DB.s` 실패 시 토스트 경고 + 미전송 변경분을 localStorage에 보관, 온라인 복귀 시 자동 재전송. 상단에 "동기화 대기 N건" 배지 | 중 |
| **P0-2. Firestore 보안 규칙 정비** | 인증된 사용자만 쓰기, `appData/_adminAuth`는 읽기 차단, 필드 검증. 콘솔에서 규칙 배포 | 소 |
| **P0-3. XSS 방어 일괄 적용** | 사용자 입력 출력부 전수 점검 후 `_esc()` 적용 | 중 |
| **P0-4. 오프라인 사진 큐 영속화** | IndexedDB에 Blob 보관, 재기동 시 이어서 업로드 | 중 |
| **P0-5. SW 캐시 보강** | `index.html`·아이콘 캐시, 스테일-와일-리밸리데이트 전략 → 산속 무신호에서도 앱 실행 보장 | 소 |

### Phase 1 — 단기 (현재 기능 완성) · 1~2주
| 항목 | 내용 | 작업량 |
|---|---|---|
| **P1-1. FCM 푸시 완성** ⭐ | VAPID 키 발급 + Cloud Functions(또는 Apps Script)로 `rescues` 생성 감지 → 전 직원 푸시 "🚨 구조 1보 발생". 재난 플랫폼의 존재 이유 | 중~대 |
| **P1-2. 팀 데이터 충돌 방지** | 팀 통과 기록을 트랜잭션 또는 팀별 서브문서로 저장, 덮어쓰기 유실 제거 | 중 |
| **P1-3. 삭제 캐스케이드** | 구조/시설 삭제 시 댓글·사진·이력 동반 정리 | 소 |
| **P1-4. 타임라인-타임테이블 통합** | `wpLog`(통과기록)·`timetable`·2보 보고를 하나의 시간순 타임라인으로 보고서에 표시 | 중 |
| **P1-5. 직원 명부 복원** | 승인 사용자(approvedUsers)를 팀원 선택 풀로 연결 — 현재 끊긴 흐름 복구 | 소 |
| **P1-6. 환자 인계 기록** | 하산/이송 완료 시 인계 대상(119·병원·헬기)·시각·인계자 입력 → 보고서 자동 반영 | 소 |

### Phase 2 — 중기 (현장 운영성 강화) · 1~2개월
| 항목 | 내용 | 작업량 |
|---|---|---|
| **P2-1. GPS 자동 통과 감지** ⭐ | 대원 휴대폰 GPS를 주기 업로드 → 다음 표지판 반경 도달 시 자동 "통과 처리할까요?" 제안. 수동 버튼 의존 탈피 | 대 |
| **P2-2. 본부 상황판 모드** | 큰 화면용: 진행중 전 구조 + 팀 위치 + 경과시간 자동 갱신 대시보드 (TV/모니터 거치) | 중 |
| **P2-3. 경과시간 표시** | 1보 접수부터 실시간 경과, 단계별(출동→조우→하산) 소요시간 — 골든타임 관리 | 소 |
| **P2-4. 사진 다중 업로드 + 갤러리** | 구조/위험/점검 사진 여러 장, 보고서 갤러리 표시 | 중 |
| **P2-5. 월별 자동 통계 리포트** | 월간 구조 통계 요약(건수·유형·구간별) 자동 생성, 인쇄용 양식 | 중 |
| **P2-6. 위험구간 히트맵** | 과거 사고 좌표 누적 → 지도 히트맵, 다발 구간 식별 | 중 |

### Phase 3 — 장기 (구조 개선·확장)
| 항목 | 내용 | 작업량 |
|---|---|---|
| **P3-1. 코드 모듈 분리** | 7,600줄 단일 파일 → 기능별 JS 분리(빌드 없이 분할 가능), 핵심 로직 단위 테스트 도입. 유지보수 비용 절감의 근본 대책 | 대 |
| **P3-2. 기상 연계 위험 예보** | 기상특보 + 과거 사고 패턴 → "호우특보 시 ○○구간 낙석 주의" 자동 알림 | 대 |
| **P3-3. 다국어 지원** | 외국인 탐방객 사고 대응용 영어 UI(신고 접수 화면 우선) | 중 |
| **P3-4. 상황 공유 채널** | 구조 건별 실시간 채팅(무전 보조) — 댓글 기능 확장 | 중 |

## 4. 권장 착수 순서

```
1주차: P0-1 (저장 재시도) + P0-2 (보안 규칙) + P0-5 (SW 캐시)   ← 데이터 유실·유출 차단
2주차: P1-1 (FCM 푸시)                                        ← 가장 체감 큰 기능
3주차: P0-3 (XSS) + P1-2 (충돌 방지) + P1-5 (직원 명부)
4주차~: P1-4, P1-6 → Phase 2 순차 진행
```

**판단 근거**: 이 앱의 본질은 "사고 발생을 빠짐없이 알리고, 팀 위치를 정확히 기록하는 것". 그래서 ① 기록이 사라지지 않게(P0-1·P1-2), ② 꺼진 폰에도 알림이 가게(P1-1), ③ 개인정보가 새지 않게(P0-2·P0-3)가 다른 모든 신규 기능보다 앞서야 함.
